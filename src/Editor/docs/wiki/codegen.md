# Generación de código

El editor dispone de **dos mecanismos** de generación de código:

1. **SourceGenerator Roslyn** (`MonoGame.Editor.SourceGenerator`): genera metadatos estáticos por escena en tiempo de compilación de `GameApp`. Sin reflexión, compatible AOT.
2. **SceneCodeGenerator** (en `MonoGame.Editor.Core/CodeGen`): genera `partial class` con el código de inicialización del `GameWorld` para integrarlo con el Kernel.

---

## SourceGenerator Roslyn (`SceneSourceGenerator`)

### Propósito

Cada `*.scene.json` declarado como `<AdditionalFiles>` en `GameApp.csproj` produce automáticamente una clase `{NombreEscena}_Scene` con constantes y metadatos de los objetos. El juego puede referenciar estas constantes para cargar escenas por nombre sin usar reflexión.

### Configuración en `GameApp.csproj`

```xml
<!-- Referencia al generador como analizador -->
<ProjectReference Include="path/to/MonoGame.Editor.SourceGenerator.csproj"
                  OutputItemType="Analyzer"
                  ReferenceOutputAssembly="false" />

<!-- Escenas del editor como archivos adicionales -->
<AdditionalFiles Include="../../../.editor/scenes/**/*.scene.json" />
```

### Ejemplo de salida generada

Para `.editor/scenes/MainMenu.scene.json`:

```csharp
// <auto-generated/>
// Source: .editor/scenes/MainMenu.scene.json

using System.Collections.Generic;

/// <summary>Auto-generated metadata for scene <c>MainMenu</c>.</summary>
public static class MainMenu_Scene
{
    /// <summary>Scene display name.</summary>
    public const string Name = "MainMenu";

    /// <summary>Relative path to the source scene JSON file.</summary>
    public const string Path = ".editor/scenes/MainMenu.scene.json";

    /// <summary>Object names defined in this scene.</summary>
    public static class ObjectNames
    {
        public const string Background = "Background";
        public const string PlayButton = "PlayButton";
    }

    /// <summary>Returns metadata for all root objects in the scene.</summary>
    public static IEnumerable<SceneObjectMetadata> GetRootObjects()
    {
        yield return new SceneObjectMetadata("Background", true, 0f, 0f, 0f);
        yield return new SceneObjectMetadata("PlayButton", true, 320f, 240f, 0f);
    }
}

/// <summary>Lightweight metadata for a scene game object.</summary>
public readonly struct SceneObjectMetadata
{
    public string Name     { get; }
    public bool   Active   { get; }
    public float  X        { get; }
    public float  Y        { get; }
    public float  Rotation { get; }
    public SceneObjectMetadata(string name, bool active, float x, float y, float rotation)
        => (Name, Active, X, Y, Rotation) = (name, active, x, y, rotation);
}
```

### Características del SourceGenerator

- **Proyecto**: `MonoGame.Editor.SourceGenerator`, target `netstandard2.0`.
- **Implementa**: `IIncrementalGenerator` (API incremental: solo regenera las escenas que cambiaron).
- **JSON parsing**: usa `System.Text.Json.JsonDocument` (disponible como NuGet en netstandard2.0).
- **Nombres únicos**: si hay dos entidades con el mismo nombre, el segundo recibe sufijo `_{hash}`.
- **Errores**: errores de parseo emiten diagnóstico `MGEDITOR001` (Warning) sin romper la compilación.
- **Archivos de salida**: `{NombreEscena}_Scene.g.cs` en la carpeta `obj/` del proyecto.

## SceneCodeGenerator (`partial class` para el Kernel)

El editor puede generar código C# que inicializa las escenas en tiempo de ejecución, siguiendo los patrones del Kernel (`Scene`, `GameWorld`, `GameEntity`).

---

### Concepto general

**Serialización**: los campos se guardan en `.editor/scenes/{nombre}.scene.json`. A partir de ese JSON, el generador produce un archivo `.Generated.cs` con una clase `partial` que inicializa el mundo del juego.

El desarrollador escribe la lógica personalizada en la otra mitad de la `partial`:

```
.editor/scenes/Gameplay.scene.json      ← Editor edita aquí
         ↓ GenerateSceneAsync()
src/GameApp/Generated/Scenes/GameplayScene.Generated.cs  ← Auto-generado, no editar
src/GameApp/Scenes/GameplayScene.cs                       ← El desarrollador escribe aquí
```

Ambos archivos forman una `partial class GameplayScene : Scene`.

---

## Ejemplo de código generado

Para una escena llamada "Gameplay" con un `Player` (con `PlayerMovementBehaviour`, `Speed=5.0`) y un `HUD_Root` con hijo `HealthBar`:

**`Generated/Scenes/GameplayScene.Generated.cs`** (generado automáticamente, no editar):
```csharp
// AUTO-GENERATED by MonoGame Editor 2026-05-27 — DO NOT EDIT MANUALLY
// Source: Editor/Scenes/Gameplay.scene.json
// Safe to commit; regenerated automatically on scene save.
using Alca.MonoGame.Kernel.ECS;
using Alca.MonoGame.Kernel.Scenes;
using Microsoft.Xna.Framework;
using MiJuego.Behaviours;

namespace MiJuego.Scenes;

public sealed partial class GameplayScene : Scene
{
    /// <inheritdoc/>
    protected override GameWorld? CreateWorld() => new GameWorld();

    /// <summary>Populates the world with all entities defined in the editor scene.</summary>
    protected override void InitializeWorld()
    {
        var world = World!;

        // ── Entity: Player ──────────────────────────────────────────────────────
        var player_0 = world.CreateEntity("Player", new Vector2(100f, 200f));
        player_0.AddTag("Damageable");
        var playerMovement_0 = player_0.AddComponent<PlayerMovementBehaviour>();
        playerMovement_0.Speed = 5f;

        // ── Entity: HUD_Root ─────────────────────────────────────────────────────
        var hudRoot_1 = world.CreateEntity("HUD_Root", Vector2.Zero);

        // ── Entity: HealthBar (hijo de HUD_Root) ─────────────────────────────────
        var healthBar_2 = world.CreateEntity("HealthBar", new Vector2(10f, 10f));
        healthBar_2.SetParent(hudRoot_1);
    }
}
```

**`Scenes/GameplayScene.cs`** (escrito una sola vez por el desarrollador, **nunca se sobreescribe**):
```csharp
namespace MiJuego.Scenes;

public sealed partial class GameplayScene : Scene
{
    public override void Initialize()
    {
        base.Initialize();
        // Lógica personalizada del desarrollador aquí
    }
}
```

## Cuándo se genera el código

Hay tres formas de lanzar la generación:

1. **Manual por escena**: menú `Project → Generate Scene Code` (o `Ctrl+G`). Genera el código de la escena activa.
2. **Manual para todas las escenas**: menú `Project → Generate All Scenes`. Itera sobre todos los `.scene.json` del proyecto.
3. **Automático al guardar**: si `ProjectSettings.GenerateOnSave = true`, se genera cada vez que se guarda la escena con `Ctrl+S`.

## Cómo funciona `SceneCodeGenerator` internamente

### 1. Validación inicial

Antes de generar, comprueba:
- Que `ProjectSettings.RootNamespace` no esté vacío.
- Que `project.GameCsprojPath` esté configurado.
- Que `project.GameSourcePath` sea una carpeta válida.

### 2. Cálculo de la ruta de salida

```
{GameSourcePath}/{GeneratedCodeFolder}/Scenes/{NombreEscena}Scene.Generated.cs
```

Ejemplo: `src/MiJuego/Generated/Scenes/GameplayScene.Generated.cs`

### 3. Hash para evitar escrituras innecesarias

El generador calcula el MD5 del contenido que va a escribir. Si el archivo ya existe y tiene el mismo hash, **no sobreescribe** (preserva la fecha de modificación y evita recompilaciones innecesarias).

### 4. Generación de usings

Los `using` se recopilan automáticamente:
- Siempre incluye: `Alca.MonoGame.Kernel.ECS`, `Alca.MonoGame.Kernel.Scenes`, `Microsoft.Xna.Framework`.
- Por cada behaviour en la escena, añade el namespace de ese tipo.
- Si algún behaviour es `SpriteRendererBehaviour`, añade `Microsoft.Xna.Framework.Graphics`.
- Si la escena usa subsistemas, añade los namespaces correspondientes (`Alca.MonoGame.Kernel.Physics`, etc.).

### 5. Generación de `CreateWorld()`

Si `WorldConfig` es `null` o no tiene subsistemas activos:
```csharp
protected override GameWorld? CreateWorld() => new GameWorld();
```

Si hay subsistemas configurados, genera un método completo:
```csharp
protected override GameWorld? CreateWorld()
{
    var world = new GameWorld();
    world.PhysicsWorld = new Physics2DWorld(new Vector2(0f, -9.8f));
    // ... otros subsistemas ...
    return world;
}
```

### 6. Generación de `InitializeWorld()`

Itera recursivamente sobre la jerarquía de `RootGameObjects`:

- Por cada entidad, genera `world.CreateEntity("Nombre", posición)`.
- Si tiene padre, genera `entity.SetParent(padreVar)`.
- Si tiene tags, genera `entity.AddTag("tag")`.
- Por cada behaviour, genera `entity.AddComponent<Tipo>()` (o manejo especial para `SpriteRendererBehaviour`).
- Por cada propiedad del behaviour con valor no-default, genera la asignación.

### 7. Resolución de colisiones de nombres de variable

Si hay dos entidades con el mismo nombre (por ejemplo dos "Enemy"), el generador añade sufijos numéricos: `enemy_0`, `enemy_1`. Usa un rastreador de nombres para garantizar unicidad.

### 8. Manejo especial de `SpriteRendererBehaviour`

`SpriteRendererBehaviour` no tiene constructor sin parámetros (requiere `Texture2D`). En lugar de `AddComponent<T>()`, el generador emite:

```csharp
var spriteRenderer_0 = new SpriteRendererBehaviour(Content.Load<Texture2D>("Textures/player"));
entity.Add(spriteRenderer_0);
spriteRenderer_0.OtraPropiedad = valor;
```

### 9. Tipos de propiedad soportados en CodeGen

| Tipo C# | Formato generado |
|---------|-----------------|
| `float` | `5f` |
| `int` | `5` |
| `bool` | `true` / `false` |
| `string` | `"texto"` |
| `Vector2` | `new Vector2(1f, 2f)` o `Vector2.Zero` |
| `Vector3` | `new Vector3(1f, 2f, 3f)` |
| `Color` | `Color.Black` o `new Color(r, g, b, a)` |
| `enum` | `NombreEnum.Valor` |

Las propiedades cuyo valor coincide con el default del tipo **se omiten** para mantener el código limpio.

### 10. Archivo stub (parte manual)

Si `{NombreEscena}Scene.cs` no existe, el generador crea un stub vacío:

```csharp
namespace MiJuego.Scenes;

public sealed partial class GameplayScene : Scene
{
}
```

Este archivo **nunca se sobreescribe** en ejecuciones posteriores. Es el espacio del desarrollador.

### 11. Añadir al `.csproj`

Si el `.csproj` del juego usa inclusiones de archivo explícitas (no wildcards), `CsprojFileEditor.EnsureFileIncludedAsync` añade la referencia al nuevo archivo. Los proyectos SDK-style modernos ya incluyen todos los `.cs` por defecto, así que en la mayoría de casos no hace falta.

## Generación de esqueleto de Behaviour

**Menú**: `Project → New Behaviour...`

El `BehaviourSkeletonGenerator` crea un archivo `.cs` con la clase lista para implementar:

```csharp
// Generated by MonoGame Editor
using Alca.MonoGame.Kernel.ECS;
using Microsoft.Xna.Framework;
using Microsoft.Xna.Framework.Graphics;  // solo si se eligió Draw

namespace MiJuego.Behaviours;

/// <summary>PlayerMovement behaviour.</summary>
public sealed class PlayerMovement : GameBehaviour
{
    public override void Awake()
    {
    }

    public override void Start()
    {
    }

    public override void Update(GameTime gameTime)
    {
    }

    public override void Draw(GameTime gameTime, SpriteBatch spriteBatch)
    {
    }

    public override void OnDestroy()
    {
    }
}
```

**Importante**: la firma de `Draw` requiere dos parámetros: `(GameTime gameTime, SpriteBatch spriteBatch)`.

## Flujo completo: de la escena al juego en ejecución

```
Editor                              Juego en runtime
──────────────────────────────      ─────────────────────────────────
.editor/scenes/Gameplay.scene.json
           │
           ├──[SourceGenerator]──► GameApp/obj/Gameplay_Scene.g.cs  (metadatos estáticos)
           │
           │ GenerateSceneAsync()
           ▼
src/GameApp/Generated/Scenes/GameplayScene.Generated.cs
           │
           │ dotnet build
           ▼
GameApp.exe
           │
           │ SceneManager.LoadScene<GameplayScene>()
           ▼
GameplayScene.CreateWorld()     ← código generado crea GameWorld con subsistemas
GameplayScene.InitializeWorld() ← código generado crea entidades y behaviours
GameplayScene.Initialize()      ← código del desarrollador
```

## Rescan de behaviours tras compilar

Al compilar el proyecto (`Project → Build Game`), si el build tiene éxito, el editor ejecuta automáticamente `GameBehaviourScanner.ScanAssemblyAsync(dllPath)` para actualizar el `GameObjectRegistry` con los nuevos tipos. Así, los nuevos behaviours aparecen inmediatamente en el diálogo de "Add Behaviour".
